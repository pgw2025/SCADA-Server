 根据对 Client 项目代码（特别是 types.ts 和 store.ts）的深度分析，该 SCADA 系统具有高度的模块化和工业标准属性。为了支撑前端的功能需求，后端数据库的设计需要涵盖设备拓扑、变量建模、数据采集、HMI
  组态、逻辑自动化以及时序数据存储。

  以下是为您设计的后端数据库表结构方案，分为六个逻辑模块：

  1. 核心资产模块 (Asset Management)
  该模块定义了工业场景的基础物理和逻辑结构。

   * Areas (区域表)
       * Id (PK), Name, Description
       * 说明：用于划分厂区、车间等。
   * DataModels (变量模型表)
       * Id (PK), Name, Description, Type (OPCUA/S7/MQTT/Virtual)
       * 说明：作为设备模板，定义一类设备的变量结构。
   * ModelVariables (模型变量定义表)
       * Id (PK), ModelId (FK), Key (变量标识), Name, Type (Analog/Digital), DataType (INT/REAL/BOOL), Unit, Min, Max, Address (寄存器地址), Description, IsStored (是否存入历史库), StoreMode
         (变化存储/周期存储).
       * 说明：定义模型下的具体点位。
   * Devices (设备表)
       * Id (PK), Name, Code (唯一编码符), AreaId (FK), ModelId (FK), Type, IpAddress, Port, Topic (MQTT主路径), Status (Online/Offline), CpuType, Rack, Slot, LastUpdated.
       * 说明：物理设备的实例。

  2. 联接与接口模块 (Connectivity)
  处理与外部协议和服务端的连接配置。

   * MqttServers (MQTT 服务器表)
       * Id (PK), Name, BrokerUrl, Port, ClientId, Username, Password, TopicPrefix.
   * DatabaseConfigs (外部数据库配置表)
       * Id (PK), Name, Type (Realtime/Historical), BackendType (MySQL/InfluxDB等), Host, Port, Username, Password, DatabaseName.
       * 说明：对应前端“数据库管理”视图，支持多库存储。
   * ExposedDataInterfaces (数据对流接口表)
       * Id (PK), Name, RouteUrl, RequestMethod, DeviceId (FK), ExposedKey, Active.
       * 说明：将内部变量暴露为标准的 RESTful API。

  3. HMI 组态模块 (Visual Logic)
  存储前端拖拽生成的界面数据。

   * ScadaProjects (组态工程表)
       * Id (PK), Name, Description, CreatedAt.
   * ScadaPages (组态页面表)
       * Id (PK), ProjectId (FK), Name, IsHome.
   * HmiComponents (HMI 控件表)
       * Id (PK), PageId (FK), Type (Pump/Valve/Gauge等), Name, X, Y, Width, Height, ZIndex, BindField (关联的变量Key), PropsJson (存储颜色、阈值、模式等 JSON 对象).

  4. 逻辑与自动化模块 (Automation)
  处理设备间的联动、脚本及定时任务。

   * DataConversions (数据联动/转换表)
       * Id (PK), Name, SourceDeviceId (FK), SourceVariableKey, TargetDeviceId (FK), TargetVariableKey, Active.
       * 说明：实现“变量 A 改变，自动写入变量 B”的逻辑。
   * VariableTriggers (告警/联动触发器表)
       * Id (PK), Name, DeviceId (FK), VariableKey, Condition (>, <, =), Threshold, ActionType (Alarm/Linkage), AlarmLevel, LinkageVariableKey, LinkageValue, Active.
   * SystemScripts (系统脚本表)
       * Id (PK), Name, Code (JavaScript 内容), TriggerType (Auto/Manual), IntervalSeconds.
   * ScheduledTasks (定时任务表)
       * Id (PK), Name, Type (Backup/SetValue/Script), CronExpression, ParamsJson, Active.

  5. 时序与日志模块 (Data & Logs)
  处理高频产生的动态数据。

   * RealtimeData (实时数据快照表 - 建议使用缓存/内存表)
       * DeviceId (PK), VariableKey (PK), Value, Timestamp.
   * HistoricalRecords (历史采样表 - 建议使用时序数据库如 TimescaleDB 或 InfluxDB)
       * Id (PK), DeviceId, VariableKey, Value, Timestamp.
   * SystemLogs (系统日志表)
       * Id (PK), Timestamp, Level (Info/Warning/Error), Source, Content.

  6. 系统与安全模块 (System)

   * SystemUsers (用户表)
       * Id (PK), Username, PasswordHash, Role (Admin/Operator/Viewer), Status.
   * SystemConfig (全局配置表 - 单行或 Key-Value 对)
       * SystemTitle, PollIntervalMs, MqttBrokerHost, RetentionPeriodDays.

  建议与核心设计考量：

   1. 变量绑定一致性：前端大量使用 bindField 字符串来映射变量。在后端设计中，VariableKey 应该在设备范围内唯一，通过 DeviceId + VariableKey 唯一定位一个数据源。
   2. JSON 字段的使用：由于 HMI 控件 (HmiComponents) 和定时任务 (ScheduledTasks) 的参数差异很大且频繁变动，建议使用关系型数据库（如 PostgreSQL/MySQL 8.0+）的 JSONB 或 JSON 字段存储 Props。
   3. 读写分离：
       * 配置数据（设备、模型、页面）存放在关系型数据库中。
       * 实时遥测值建议存放在 Redis 中，以支持前端 SignalR 的高频推播。
       * 历史趋势数据应存放在时序数据库（TSDB），以应对前端“历史查询”视图中的大数据量聚合需求。

        根据 Client 项目的结构（特别是 store.ts 中的 axios 调用、SignalR 集成以及各业务组件的逻辑），我为您设计了一套工业级后端 API 架构。

  后端建议采用 RESTful API 处理配置类业务，配合 SignalR (WebSocket) 处理高频实时遥测数据。

  ---

  1. 核心实时通信 (SignalR Hub)
  前端代码中定义了 /hubs/scada，这是整个系统的“神经中枢”。

   * Hub 名称: ScadaHub
   * 服务端推送到客户端 (Methods):
       * ReceiveVariableUpdate(string variableKey, object value): 当 PLC 或传感器值变化时，实时推送到前端。
       * ReceiveSystemLog(SystemLog log): 实时推送系统生成的警报或日志。
       * ReceiveDeviceStatus(string deviceId, string status): 设备上线/下线通知。
   * 客户端调用服务端 (Methods):
       * WritePlcVariable(string variableKey, object newValue): 前端点击按钮或开关时，直接通过长连接下发控制指令。

  2. REST API 接口设计

  A. 设备与资产管理 (Asset Management)
   * GET /api/devices: 获取所有设备列表（含当前实时值快照）。
   * GET /api/devices/{id}: 获取单个设备详情。
   * POST/PUT/DELETE /api/devices: 设备的增删改。
   * GET /api/areas: 获取区域列表。
   * GET /api/data-models: 获取变量模型（OPCUA/S7 等模板）。

  B. 数据采集与控制 (Data Acquisition & Control)
   * POST /api/variables/write: 备用写入接口（当 SignalR 不可用时使用）。
       * Payload: { "deviceId": "dev-1", "variableKey": "pump_state", "value": true }
   * GET /api/history: 获取历史时序数据。
       * Query: ?variableKey=temp&startTime=...&endTime=...&limit=100
       * 用途: 用于渲染“历史查询”视图中的 ECharts 曲线。

  C. HMI 组态编辑器 (HMI Editor)
   * GET /api/scada/projects: 获取所有组态工程。
   * GET /api/scada/projects/{id}/pages: 获取工程下的所有页面及控件。
   * PUT /api/scada/pages/{id}/components: 核心接口。保存前端拖拽后的布局。
       * Payload: HMIComponent[] (包含坐标、绑定关系、样式 JSON)。

  D. 自动化逻辑 (Automation & Logic)
   * GET/POST /api/triggers: 变量触发器（告警阈值）管理。
   * GET/POST /api/tasks: 定时任务（Cron）管理。
   * POST /api/tasks/{id}/execute: 手动强制触发一个定时任务（如手动备份）。
   * GET/POST /api/scripts: 系统脚本（JS Sandbox）管理与运行。

  E. 系统监控与配置 (System & Config)
   * GET /api/sys/status: 获取服务器资源占用（CPU/内存/磁盘），对应前端 Dashboard 顶部的仪表盘。
   * GET /api/sys/logs: 分页获取历史日志。
   * GET /api/sys/config: 获取/更新系统全局设置（如轮询间隔、MQTT 地址）。
   * GET /api/database/configs: 外部数据库连接状态检查与配置。

  ---

  3. 核心业务流程 (Sequence)

  控制指令流 (Write Flow):
   1. 用户点击 HMIButton.vue。
   2. 前端 调用 store.ts 的 setDeviceVariableValue。
   3. 后端 SignalR Hub 接收到 WritePlcVariable。
   4. 后端驱动层 根据变量地址 (如 DB10.DBX24.0) 调用 S7 或 OPCUA 驱动写入物理设备。
   5. 设备反馈 成功后，后端再次通过 ReceiveVariableUpdate 广播给所有在线客户端。

  数据采集流 (Polling/Sub Flow):
   1. 后端采集服务 (Background Service) 按照 pollIntervalMs 轮询 PLC。
   2. 后端时序模块 判断 isStored，若是则写入 InfluxDB/ClickHouse。
   3. 后端触发器引擎 判断是否触发 VariableTrigger。
   4. 后端推送 变化的值到前端，前端组件通过 computed 自动更新 UI 状态。

  4. 技术栈推荐 (匹配 Client 设计)

   * 框架: ASP.NET Core 8.0 (原生支持 SignalR，工业界标准)。
   * 通信驱动:
       * S7NetPlus: 用于西门子 PLC 通讯。
       * OpcLabs.QuickOpc 或 Opc.UaFx: 用于 OPC UA 服务。
       * MQTTnet: 用于 MQTT 集成。
   * 缓存: Redis (存储“实时数据快照”，避免频繁查库)。
   * 任务调度: Quartz.NET 或 Hangfire (实现 ScheduledTask 逻辑)。
   * 安全: JWT 认证，配合前端 performLogin 逻辑。

  这个设计完全对应了您 Client 项目中的 DashboardView, DeviceManagementView, ScadaTopologyView 等所有 UI 模块。您需要先从设备管理和实时数据推送这两个接口开始开发。
